# Excel 数据流引擎 — 商业化分析

> 编写时间：2026-03-19
> 基于项目当前状态（纯前端、开源、React + React Flow + Web Worker）的客观分析

---

## 一、项目定位与竞品格局

### 1.1 项目核心能力

- 纯前端、零部署、零依赖，浏览器直接使用
- 可视化拖拽 DAG 编排数据处理流程
- 内置业务算子：阶梯规则（V1/V2）、条件赋值（多条件+列引用）、countif 聚合等
- 模板导入导出，流程可复用可交接
- 数据不出本地，天然满足隐私合规

### 1.2 竞品对比

| 维度 | 本项目 | Dataptic | Flowfile | Power Query | KNIME | Alteryx |
|------|--------|----------|----------|-------------|-------|---------|
| 部署方式 | 纯前端浏览器 | 纯前端浏览器 | Python + Web UI | 内置于 Excel | 桌面应用 (Java) | 桌面应用 |
| 费用 | 免费开源 | 免费开源 | 免费开源 | 随 Office 免费 | 桌面免费 | ~$5,195/年/用户 |
| 拖拽节点编排 | 是 | 是 | 是 | 否（菜单式线性步骤） | 是 | 是 |
| DAG 多分支 | 是 | 是 | 是 | 否（线性） | 是 | 是 |
| 算子/变换数量 | ~10 | ~10 | 30+ | 300+ | 4000+ | 200+ |
| 业务算子（阶梯/条件赋值） | 内置 | 无 | 需写 Python | 需写 M 语言 | 需组合多节点 | 需 Formula 工具 |
| 数据量上限 | ~10万行 | 受浏览器限制 | 百万行+ (Polars) | 百万行+ | 千万行+ | 千万行+ |
| 实时预览 | 执行后 | 逐节点实时 | 是 | 逐步 | 是 | 是 |
| 模板复用 | JSON 导入导出 | 无 | YAML/JSON | 可复制查询 | 工作流导出 | .yxmd 导出 |
| 数据隐私 | 纯本地 | 纯本地 | 本地 | 本地/云端 | 本地 | 本地 |

### 1.3 差异化优势

- **vs Power Query**：可视化 DAG 编排体验远胜线性步骤、业务算子开箱即用、零安装
- **vs KNIME/Alteryx**：零门槛、免费、轻量、业务算子直接可用（无需组合或编程）
- **vs Flowfile**：不需要 Python 环境、业务算子内置、更低使用门槛
- **vs Dataptic**：业务算子丰富得多（阶梯规则、条件赋值、countif 聚合）、模板系统

### 1.4 差异化劣势

- 算子数量是最大短板（~10 vs 竞品几十到几千）
- 定位越通用，竞争越激烈
- 不是实时预览（需手动执行）
- 大数据量能力受浏览器限制

## 二、商业价值评估

### 2.1 解决的真实痛点

薪资结算、分润计算是大量中小企业（MCN/公会、电商、外包）的刚需。目前普遍靠"Excel 高手 + 复杂公式 + 手动操作"完成，存在：

- 规则变更时改公式容易出错
- 人员离职导致"只有他会算"
- 每月重复劳动，无法复用

本工具把隐性的 Excel 公式逻辑变成可视化、可复用、可交接的流程。

### 2.2 客观判断

| 维度 | 评估 |
|------|------|
| 作为开源工具/个人作品 | **有价值** — 解决实际问题，展示技术能力 |
| 作为独立产品商业化 | **困难** — 市场小、替代多、付费意愿低 |
| 作为垂直 SaaS 的功能模块 | **有潜力** — 嵌入 MCN 管理系统/公会 ERP |
| 作为咨询/定制服务的交付工具 | **可行** — 帮客户搭建结算流程，收服务费 |

### 2.3 市场风险

- 需要"用 Excel 做复杂结算"的企业量级有限，目标客户（中小 MCN、公会、外包）付费意愿和能力有限
- Excel 本身（VBA/Power Query）、Python pandas、AI 辅助写公式都是替代方案
- 纯前端 + 开源 = 用户迁移成本为零，无粘性壁垒

## 三、商业模式分析

### 3.1 前端付费墙（不可行）

**结论：技术原理上走不通。**

原因：

1. **计算必须在明文上进行** — JavaScript 无法对加密数据做 sum、filter、条件比较等运算。要计算就必须解密，解密后数据在内存中以明文存在
2. **浏览器环境无法保护代码** — DevTools 可以拦截 Worker 消息、直接从 Zustand store 取完整明文数据
3. **前端权限检查可被绕过** — 任何 `if (!user.isPaid)` 检查都可以在控制台修改
4. **同态加密不现实** — 理论上可以对加密数据做计算，但性能开销 10000 倍以上，实际业务不可接受

加密方案的漏洞示意：

```
输入 Excel → [必须明文解析] → [必须明文计算] → [结果已在内存中]
                                                      ↓
                                              用户可通过 DevTools 获取
                                              加密导出环节已无意义
```

### 3.2 开源核心 + 付费云服务（推荐，最主流）

**模式**：工具开源免费 → 托管云版本提供增值服务

云版本增值点：

- **团队协作**：多人共享模板、流程版本管理
- **云端存储**：模板/规则/历史结果自动保存
- **更大数据量**：服务端执行（Python/Polars），突破浏览器 10 万行限制
- **定时执行**：每月自动跑结算流程
- **审计日志**：谁在什么时间用什么模板跑了什么数据

参考案例：Grafana、GitLab、Supabase 等开源项目的经典模式。

| 优点 | 缺点 |
|------|------|
| 可规模化，SaaS 模式有持续收入 | 需要建设后端，工程投入大 |
| 开源版本保持纯前端不变 | 运维成本 |
| 数据量和性能可以大幅提升 | 需要用户信任云端数据安全 |

### 3.3 轻后端付费方案（折中方案）

**模式**：前端免费编排 + 少量预览 → 付费在服务端执行全量计算

```
前端（免费）                          轻后端（付费控制）
┌─────────────────────┐              ┌──────────────────────┐
│ 拖拽编排流程          │              │ 接收：流程JSON + Excel │
│ 配置算子参数          │              │ 服务端执行完整 DAG     │
│ 上传 Excel           │──付费执行──→ │ 验证付费状态           │
│ 预览前 5 行（免费）   │              │ 返回完整结果/下载链接   │
└─────────────────────┘              └──────────────────────┘
```

- 免费用户：可以编排流程、上传数据、预览前 5 行结果
- 付费用户：全量执行 + 下载在服务端完成，无法绕过
- 后端可以很轻量：Node.js/Python 服务 + Supabase/Clerk 认证

| 优点 | 缺点 |
|------|------|
| 改动量相对小 | 用户数据需上传服务端，影响隐私卖点 |
| 付费验证在服务端，无法绕过 | 需要服务器/Serverless 成本 |
| 前端代码仍可开源 | |

### 3.4 模板/场景市场（轻量级）

**模式**：工具免费，预制行业结算模板付费

- 售卖经过验证的行业模板：MCN 主播结算、电商佣金分润、外包薪资计算等
- 模板是 JSON 文件，包含业务 know-how（阶梯规则配置、条件编排、流程设计）
- 类似 Notion 模板市场、Figma 社区模板

| 优点 | 缺点 |
|------|------|
| 不需要后端，不需要改架构 | JSON 文件售出后可被转发，无技术保护 |
| 积累行业经验和口碑 | 收入规模有限 |
| 轻投入，可快速验证市场 | 模板通用性有限，行业差异大 |

### 3.5 企业定制 + 技术支持（最务实）

**模式**：开源引流 → 企业客户 → 定制开发 + 年度技术支持

- 开源版本作为获客工具
- 企业客户需求：定制算子、对接内部系统、私有部署、培训
- 收费：一次性定制费 + 年度维护费

| 优点 | 缺点 |
|------|------|
| 不需要改架构，利润率高 | 不可规模化，本质是做服务 |
| 可以深入了解行业需求 | 受限于团队人力 |
| 客户粘性高（定制后迁移成本大） | |

### 3.6 SDK 嵌入授权

**模式**：把数据流引擎做成可嵌入的 React 组件/SDK，授权给其他 SaaS 产品

- 目标客户：MCN 管理平台、HR 系统、电商 ERP 等需要"结算模块"的产品
- 按许可证收费
- 类似 Stripe 做支付、TinyMCE 做富文本编辑器

| 优点 | 缺点 |
|------|------|
| 可规模化，B2B 模式 | 需要大量工程投入改造为可嵌入组件 |
| 技术壁垒相对高 | 市场教育成本高 |

## 四、综合评估与建议路径

### 4.1 可行性排序

| 模式 | 可行性 | 收入潜力 | 所需投入 | 建议优先级 |
|------|--------|---------|---------|-----------|
| 前端付费墙 | 低（技术不可行） | — | — | 不推荐 |
| 企业定制服务 | 高 | 中 | 低 | **第 1 步** |
| 模板市场 | 中 | 低 | 低 | **第 1 步（并行）** |
| 轻后端付费 | 高 | 中 | 中 | **第 2 步** |
| 开源 + 付费云服务 | 高 | 中高 | 高 | **第 3 步** |
| SDK 嵌入授权 | 中 | 高 | 很高 | 远期 |

### 4.2 推荐路径

```
阶段 1（当前）
  开源发布 + 建立行业影响力
  尝试卖模板包（低成本验证市场）
  找 2-3 个企业客户做定制（验证需求 + 赚现金流）
      ↓
阶段 2（验证后）
  加轻后端：用户认证 + 服务端执行
  免费编排预览 + 付费全量执行下载
  引入团队协作和云端存储
      ↓
阶段 3（规模化）
  完整 SaaS 云平台
  SDK 授权给其他产品
  行业模板生态
```

核心原则：**先用最小成本验证市场需求，再投入建设基础设施。不要在没有付费客户之前就建后端。**

## 五、技术演进方向（支撑商业化）

如果商业化推进，以下技术改进按优先级排序：

1. **实时预览（逐节点流式返回）** — 体验提升最明显，改动最小
2. **算子扩展**（字符串处理、日期计算、VLOOKUP 等）— 扩大适用场景
3. **更大数据量支持** — 可选 WebAssembly 或服务端执行
4. **组件化/SDK 化** — 支持嵌入到其他系统
5. **用户认证 + 轻后端** — 支撑付费模式
6. **团队协作** — 模板共享、版本管理、权限控制
